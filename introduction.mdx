Despia is a mobile capability runtime that lets web apps access native device features and publish to the App Store and Google Play without writing Swift or Java.

<Note>
  **Despia is built for builders who want the flexibility of the web with the power of native development - and ship to iOS and Android without a CLI, terminal, or complex build setup. App Store and Google Play submission is handled entirely from a browser.**
</Note>

The runtime runs on WKWebView (iOS) and the Chromium-based WebView (Android) - the same rendering engines that power Safari and Chrome - with hardware acceleration and JIT compilation enabled. Native capabilities including secure storage, biometrics, background GPS, push notifications, in-app purchases, attribution, and offline media caching are implemented in Swift and Java and exposed to JavaScript through a typed SDK. The underlying native projects can be exported and modified at any time.

## How it works

<Tabs>
  <Tab title="Developer experience">
    The `despia-native` npm package exposes a single async function. Import it and call native capabilities directly - no initialization, no configuration, no setup.

    ```javascript
    import despia from 'despia-native';
    
    // Fire and forget
    despia('successhaptic://');
    
    // With a response
    const device = await despia('get-uuid://', ['uuid']);
    console.log(device.uuid);
    ```

    The SDK handles TypeScript definitions, command queuing, promise-based variable watching with a 30-second timeout, and error handling.

  <Tab title="Internals">
    Calling `despia()` sets `window.despia` to the scheme string, which the runtime maps to `window.location.href`. On iOS, the navigation is intercepted in `WebViewController.swift` via `decidePolicyFor navigationAction`. On Android, it is intercepted in `MainActivity.java` via `shouldOverrideUrlLoading`. Results are written back to the WebView context as named window variables.

    ```javascript
    // despia('get-uuid://', ['uuid'])
    // 1. window.despia = 'get-uuid://'        set by despia-native npm
    // 2. window.location.href = 'get-uuid://' intercepted by native layer
    // 3. window.uuid = '<device-uuid>'        written back to WebView
    const device = await despia('get-uuid://', ['uuid']);
    console.log(device.uuid);
    ```

    <AccordionGroup>
      <Accordion title="Why scheme://">
        `scheme://` interception was the standard pattern for hybrid app native bridges over a decade ago, when Despia's runtime was first built. It remains in use intentionally, because it provides a set of properties that are difficult to replicate with more complex messaging systems.

        - **Simple.** A command in, a result out.
        - **Predictable.** Consistent behavior across iOS and Android.
        - **Readable.** Commands are understandable and debuggable without tooling.
        - **Searchable.** Trivial to locate across large codebases, including by AI coding tools.

        The runtime itself is fully modernized. Native capabilities are implemented in current Swift and Java. The SDK provides typed async APIs with command queuing and error handling. Data-heavy operations are routed through the on-device HTTP server at `http://localhost` where appropriate.
      </Accordion>

      <Accordion title="Why not JSON-based bridges">
        Structured JSON messaging was evaluated. In practice, `scheme://` performed better in real-world usage - particularly with AI-assisted development. AI generates flat command strings more reliably than nested JSON payloads, with less risk of malformed structures or schema mismatches. Commands are also easier to trace and reason about across a codebase. The practical result is faster iteration, fewer errors, and more predictable code generation.
      </Accordion>

      <Accordion title="localhost as a secure origin">
        For operations that do not fit the scheme pattern - file uploads, streaming, anything that benefits from standard HTTP - Despia routes through an on-device HTTP server at `http://localhost`. Because `http://localhost` is a real secure origin in both WKWebView and Chromium, Service Workers, IndexedDB, Web Crypto, and standard URL routing work without modification. No Blob URL workarounds, no Base64 encoding, no arbitrary storage quotas.
      </Accordion>
    </AccordionGroup>
  </Tab>
</Tabs>

## Core capabilities

### Identity Vault

Encrypted key-value storage backed by iCloud KV Storage on iOS and Android App Backup on Android. Values persist across uninstall and reinstall cycles. Biometric authentication is configured per key.

```javascript
// Store a value
await despia('setvault://?key=userId&value=user123&locked=false')

// Read it back
const data = await despia('readvault://?key=userId', ['userId'])
console.log(data.userId)
```

Setting `locked=true` requires Face ID or Touch ID before the value is returned. The biometric flow, fallback handling, and error states are managed by the native layer.

```javascript
await despia('setvault://?key=sessionToken&value=abc123&locked=true')

// Triggers native biometric prompt - resolves only on successful authentication
const token = await despia('readvault://?key=sessionToken', ['sessionToken'])
```

### GPS location

Real-time location tracking with configurable time intervals, distance thresholds, live frontend callbacks, optional server delivery on each point, and session replay on app reopen. Background tracking is supported on both platforms including Samsung, Huawei, Xiaomi, and OnePlus.

```javascript
window.onLocationChange = function(data) {
    if (!data.active) {
        finalizeRoute()
        return
    }
    drawPointOnMap(data.latitude, data.longitude)

    if (data.speed > 0) {
        const pace = (1000 / data.speed / 60).toFixed(2)
        updatePaceDisplay(pace + ' min/km')
    }

    if (data.horizontalAccuracy > 10) return
}

// Updates on every 1 metre of movement, with a 60-second time-based fallback
despia('location://?buffer=60&movement=100')
```

Stop tracking and retrieve the session for post-processing.

```javascript
const data      = await despia('stoplocation://', ['locationSession'])
const locations = data.locationSession

const drain = locations[0]?.battery - locations[locations.length - 1]?.battery
console.log(`Battery used: ${drain}%`)
```

## Monetization

Native App Store and Google Play billing through RevenueCat. Purchases, paywalls, entitlement checks, and subscription restore are handled by the native RevenueCat SDK. No billing SDK configuration is required on the web layer.

<Info>
  RevenueCat must be enabled in **Despia \> App \> Settings \> Integrations \> RevenueCat** and your API key configured before purchases will work.
</Info>

```javascript
const isDespia = navigator.userAgent.toLowerCase().includes('despia')

if (isDespia) {
    despia(`revenuecat://launchPaywall?external_id=${userId}&offering=default`)
} else {
    window.location.href = `https://pay.rev.cat/<your_token>/${encodeURIComponent(userId)}`
}
```

Check entitlements on app load and after each purchase.

```javascript
async function checkEntitlements() {
    const data   = await despia('getpurchasehistory://', ['restoredData'])
    const active = (data.restoredData ?? []).filter(p => p.isActive)

    if (active.some(p => p.entitlementId === 'premium')) unlockPremium()
}

checkEntitlements()
window.onRevenueCatPurchase = checkEntitlements
```

## Analytics and attribution

AppsFlyer attribution, deep linking, and event tracking. Attribution data is recorded at install time and injected into the web layer on every page load.

<Info>
  AppsFlyer must be enabled in **Despia \> App \> Settings \> Integrations \> AppsFlyer** and your AppsFlyer dev key configured before data will flow. Currently available on iOS, with Android coming soon.
</Info>

```javascript
// Available on every page load
const attribution = despia.appsFlyerAttribution
const referrer    = despia.appsFlyerReferrer     // e.g. "tiktok_ad", "organic"
const uid         = despia.appsFlyerUID
```

Emails and phone numbers are hashed with SHA-256 before transmission.

```javascript
despia("appsflyer://set_user_id?customer_user_id=" + encodeURIComponent(userId))
```

## Offline media caching

<Info>
  Local CDN is currently in final beta. To request access, email [offlinemode@despia.com](mailto:offlinemode@despia.com)
</Info>

Since PhoneGap launched in 2009, every hybrid framework has struggled with offline media. The problem is structural: files have to pass through JavaScript, which means Base64 encoding, Blob URL conversions, and memory usage that scales with file size. A 500MB video in Capacitor peaks at approximately 1.6GB of JS heap. Large files crash apps.

After 7,500 apps built on Despia's predecessor runtime, this pattern was consistent enough to justify solving properly. Local CDN bypasses JavaScript entirely for file operations. Downloads go directly to the native file system via NSURLSession (iOS) and WorkManager (Android). Files stream back through the on-device HTTP server at `http://localhost` to the hardware decoder. Memory usage is O(1) regardless of file size.

Downloads continue when the app is backgrounded or closed. On iOS, the native download progress appears in the standard system download activity. On Android, it shows in the native download manager. Both platforms handle retry on network failure automatically. When a download completes, an optional push notification fires. When the app reopens, `window.contentServerChange(item)` is called for each download that completed while the app was not running.

```javascript
window.contentServerChange = (item) => {
    videoElement.src = item.local_cdn   // standard HTTP URL, plays immediately
}

// Fire and forget - download continues in background
despia(
    `localcdn://write?url=${remoteUrl}&filename=videos/course.mp4&index=course_ep1&push=true&pushmessage="Episode 1 ready for offline viewing"`
)
```

Read cached files by index and use the `local_cdn` URL in any standard web API.

```javascript
const data = await despia(
    `localcdn://read?index=${encodeURIComponent(JSON.stringify(['course_ep1']))}`,
    ['cdnItems']
)

videoElement.src = data.cdnItems[0].local_cdn
```

|  | Capacitor | Despia Local CDN |
| --- | --- | --- |
| Peak JS memory (500 MB file) | ~1,666 MB | ~100 bytes |
| Storage limit | 50-100 MB | Full device capacity |
| Background downloads | No | Yes |
| Playback | Software decoded | Hardware accelerated |

[Local CDN reference](https://setup.despia.com/local-cdn/introduction)

### Publishing

Despia handles the full submission pipeline from a browser with no CLI, no local toolchain, and no manual credential management. Toggle capabilities, configure runtime settings, manage iOS targets (App Clips, Share Extensions, Home Widgets), and submit. Code signing, provisioning profiles, and store submission are automated through the built-in CI/CD pipeline.

For teams that want full control, native projects can be exported at any time as complete Xcode and Android Studio projects including all Swift and Java source. The Despia editor and the self-managed path are not mutually exclusive.

Web content changes do not require a new binary. Only changes to native configuration or runtime behavior trigger a new build.

## Extending the runtime

The bridge pattern is consistent across both platforms. To add a custom native capability, intercept a new scheme in `WebViewController.swift` (iOS) or `MainActivity.java` (Android) and write the result back to the WebView context. The SDK resolves custom bridges identically to built-in capabilities.

<Steps>
  <Step title="Add the native handler" icon="code">
    Intercept the new scheme in the native layer and write the result back to the WebView context.

    <CodeGroup>

    ```swift iOS - WebViewController.swift
    if requestURL.absoluteString.hasPrefix("mycustom://") {
        let result = runMyNativeCode()
        webView.evaluateJavaScript("window.myResult = '\(result)';")
        decisionHandler(.cancel)
        return
    }
    ```

    ```java Android - MainActivity.java
    if (url.startsWith("mycustom://")) {
        String result = runMyNativeCode();
        webView.evaluateJavascript("window.myResult = '" + result + "';", null);
        return true;
    }
    ```

    </CodeGroup>
  </Step>
  <Step title="Call it from your web app" icon="globe">
    Use the `despia-native` SDK exactly as you would any built-in capability.

    ```javascript
    import despia from 'despia-native';
    
    const data = await despia('mycustom://', ['myResult'])
    console.log(data.myResult)
    ```
  </Step>
</Steps>

<Tip>

## Rendering

Despia uses WKWebView on iOS and the Chromium-based WebView on Android. Both engines use hardware-accelerated compositing and JIT-compiled JavaScript. CSS animations run on the compositor thread. WebGL and Canvas 2D are GPU-accelerated. The rendering pipeline is equivalent to mobile Safari and Chrome respectively.

For requirements that cannot be met by the web platform - highly custom gesture recognizers, deeply platform-specific UI components - the native export and custom bridge paths are available.

## Works with any tool or framework

Despia is a capability layer with no opinion on how your UI is built, what state management you use, or which tools generated your code.

<CardGroup cols={2}>
  <Card title="Web frameworks" icon="code">
    React, Vue, Angular, Svelte, Next.js, Nuxt, SvelteKit, Astro, Remix, vanilla JavaScript
  </Card>

  <Card title="AI coding tools" icon="sparkles">
    Cursor, Claude Code, Lovable, v0, Base44, or any tool that generates web code
  </Card>
</CardGroup>

The integration is a single import and a function call. AI coding tools generate `despia()` calls reliably because the API surface is a flat string command with no nested configuration objects, no provider wrappers, and no initialization sequences.

```javascript
import despia from 'despia-native';

// Works in any framework, any file, any component
await despia('setvault://?key=userId&value=user123&locked=false')
despia('successhaptic://')
```

The practical model is a clean separation: your framework or AI builder handles UI, routing, and business logic. Despia handles device access, native capabilities, and store submission.

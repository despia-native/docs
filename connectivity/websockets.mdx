---

Open a WebSocket through the native runtime. Inbound messages are persisted to disk before your handler runs and replayed until acknowledged. Outbound sends queue while offline and flush in order on reconnect. Drive the connection with `websocket://` schemes, receive events through `window.onWebSocketEvent`.

<Info>
  Delivery is at-least-once, dedupe on `message_id` in your handler. Background coverage is best-effort, embed a resume cursor in your subscribe frame so the server can replay missed events.
</Info>

## Scheme reference

Five `websocket://` schemes drive the native side. Every command accepts an optional `rid` echoed back on the matching `response` event for async correlation. Always wrap dynamic values in `encodeURIComponent()`, the query is parsed `application/x-www-form-urlencoded`.

| scheme | required params | optional params | purpose |
| :-- | :-- | :-- | :-- |
| `websocket://connect` | `id`, `url` | `headers`, `protocols`, `reconnect`, `rid` | Open or re-attach a connection |
| `websocket://send` | `id`, `payload` | `rid` | Send a text frame, queued if offline |
| `websocket://subscribe` | `id` | `frame`, `rid` | Register a frame replayed on every connect |
| `websocket://disconnect` | `id` | `rid` | Close and disable auto-reconnect |
| `websocket://status` | `id` | `rid` | Get connection state as a `response` event |

## How it works

Assign your event handler, then call `websocket://connect` with a label and URL. The handler can be assigned at any point, messages buffered before assignment flush as soon as it is wired up.

```javascript
const isDespia = navigator.userAgent.toLowerCase().includes('despia')

if (isDespia) {
    window.onWebSocketEvent = function (evt) {
        if (evt.type === 'message') {
            console.log(`[${evt.id}] ${evt.payload}`)
        }
    }

    const url = encodeURIComponent('wss://example.com/ws')
    despia(`websocket://connect?id=orders&url=${url}`)
}
```

The `id` identifies the connection and is echoed back on every event for routing. Calling `connect` again on an open id is a no-op. Wrap every dynamic value passed to a `websocket://` scheme in `encodeURIComponent()`.

Right after `connect` the runtime fires a `connecting` event followed by `open` once the handshake completes.

```json
{
    "id": "orders",
    "type": "connecting",
    "ts": 1747600000,
    "attempt": 0
}
```

```json
{
    "id": "orders",
    "type": "open",
    "ts": 1747600001,
    "protocol": "graphql-transport-ws"
}
```

`protocol` is the negotiated subprotocol from the handshake, or `""` if none was requested.

## Authenticated handshakes and subprotocols

Pass HTTP handshake headers as a JSON object and subprotocols as a comma-separated list. Both go through `encodeURIComponent`.

```javascript
const url       = encodeURIComponent('wss://example.com/ws')
const headers   = encodeURIComponent(JSON.stringify({ Authorization: 'Bearer ' + token }))
const protocols = encodeURIComponent('graphql-transport-ws,graphql-ws')

despia(`websocket://connect?id=orders&url=${url}&headers=${headers}&protocols=${protocols}`)
```

Set `reconnect=false` if you want a one-shot connection that does not auto-recover, the default is `true`.

## Receiving messages

Every inbound frame arrives as a `message` event with a unique `message_id`, the frame type, the payload, and the receive timestamp. This is the only event type that participates in the acknowledgement flow.

The shape of `payload` depends on `dataType`. A text frame whose body is a JSON object or array is delivered as `dataType: "json"` with the payload already parsed, so use it directly and never call `JSON.parse` on it.

```json
{
    "id": "orders",
    "type": "message",
    "ts": 1747600042,
    "message_id": "orders:128",
    "dataType": "json",
    "payload": {
        "orderId": "ord_abc",
        "status": "filled"
    }
}
```

Plain text frames and bare JSON scalars (`42`, `"hi"`, `true`, `null`) come through as `dataType: "text"`. Scalars stay as text so an intended string `"42"` is never confused with the number `42`.

```json
{
    "id": "orders",
    "type": "message",
    "ts": 1747600043,
    "message_id": "orders:129",
    "dataType": "text",
    "payload": "pong"
}
```

Binary frames come through as `dataType: "binary"` with a base64-encoded payload. Decode with `atob` and a `Uint8Array` before use.

```json
{
    "id": "orders",
    "type": "message",
    "ts": 1747600044,
    "message_id": "orders:130",
    "dataType": "binary",
    "payload": "SGVsbG8gd29ybGQ="
}
```

`message_id` is unique per connection and monotonic, format is `<id>:<n>`. Use it as the dedupe key, the runtime can deliver the same message more than once.

## Resume on reconnect

`websocket://subscribe` registers a frame that is re-sent on every connect. Embed your cursor or last-seen id so the server can replay events that arrived while the socket was down.

```javascript
const frame = encodeURIComponent(JSON.stringify({
    action: 'subscribe',
    topics: ['orders'],
    since: lastCursor
}))

despia(`websocket://subscribe?id=orders&frame=${frame}`)
```

Only one frame is stored per connection, the latest registration replaces the previous one. Omit `frame` to clear it. Update it as your cursor advances.

The frame is sent to the server first on every connect, before any queued outbound sends or inbound replay. There is no JS-side event for the send itself, the next `message` events from your server are the proof it was received.

## Sending messages

`websocket://send` writes a text frame. If the socket is down the send persists to disk and flushes FIFO when the connection comes back. Permanent failure after 25 retries surfaces as a `send_failed` event carrying the outbound id.

```javascript
const payload = encodeURIComponent(JSON.stringify({ ping: 1 }))
despia(`websocket://send?id=orders&payload=${payload}`)
```

For binary frames, base64 encode the bytes and pass the resulting string as `payload`.

When a send permanently fails, `send_failed` fires once for that outbound and the rest of the queue keeps flowing past it.

```json
{
    "id": "orders",
    "type": "send_failed",
    "ts": 1747600099,
    "oid": "orders#42"
}
```

`oid` is the outbound id assigned by the runtime when you called `send`, format is `<id>#<n>`. Keep your own mapping if you need to correlate `send_failed` back to a specific application-level intent.

## Acknowledgement

Your handler's return value controls acknowledgement for `message` events. A return that is not `false`, or a Promise that resolves to anything other than `false`, acknowledges the message and removes it from the durable store. A return of `false`, a thrown error, or a rejected Promise forces replay.

```javascript
const seen = new Set()

window.onWebSocketEvent = function (evt) {
    if (evt.type !== 'message') return

    if (seen.has(evt.message_id)) return true

    const data =
        evt.dataType === 'json'   ? evt.payload                                          // already parsed
      : evt.dataType === 'binary' ? Uint8Array.from(atob(evt.payload), c => c.charCodeAt(0))
      :                             evt.payload                                          // raw text

    return handleBusinessEvent(evt.id, data)
        .then(() => { seen.add(evt.message_id); return true })
        .catch(err => { console.error(err); return false })
}
```

Replay triggers are WebView reload, app resume, socket reconnect, and network restore. The same `message_id` can arrive more than once, dedupe on it. For dedupe across reloads, back the set with IndexedDB or another persistent store.

## Lifecycle and error events

Route non-message events by `evt.type` for logging, UI state, and resync triggers. None of these participate in the acknowledgement flow.

```javascript
window.onWebSocketEvent = function (evt) {
    switch (evt.type) {
        case 'open':
            console.log(`[${evt.id}] connected ${evt.protocol || ''}`)
            return

        case 'reconnecting':
            console.warn(`[${evt.id}] retry #${evt.attempt} in ${evt.delay.toFixed(1)}s`)
            return

        case 'closed':
            console.warn(`[${evt.id}] closed code=${evt.code} clean=${evt.clean}`)
            return

        case 'dropped':
            resync(evt.id)
            return

        case 'send_failed':
            console.error(`[${evt.id}] outbound ${evt.oid} permanently failed`)
            return
    }
}
```

`reconnecting` fires when the runtime is waiting before the next reconnect attempt. `attempt` is the 1-indexed retry number, `delay` is the wait in seconds.

```json
{
    "id": "orders",
    "type": "reconnecting",
    "ts": 1747600055,
    "attempt": 3,
    "delay": 4.2
}
```

`closed` fires when the socket terminates. `code` is the WebSocket close code per RFC 6455, `reason` is the server-supplied reason string (often empty), `clean` is `true` only for clean server-initiated closes (code 1000).

```json
{
    "id": "orders",
    "type": "closed",
    "ts": 1747600060,
    "code": 1006,
    "reason": "",
    "clean": false
}
```

`error` fires on transient socket errors during a session and is typically followed by a `closed` and a `reconnecting` if `reconnect` is true. Most apps can ignore `error` and react only to `closed` and `reconnecting`.

```json
{
    "id": "orders",
    "type": "error",
    "ts": 1747600059,
    "error": "The network connection was lost."
}
```

`dropped` fires when inbound records are purged because they exceeded the 7-day age limit or the 5000-entry cap. Re-register your subscribe frame with a fresh cursor to replay from the server.

```json
{
    "id": "orders",
    "type": "dropped",
    "ts": 1747600100,
    "count": 17,
    "reason": "expired_or_overflow"
}
```

## Connection status

`websocket://status` returns the connection state as a `response` event. Pass a `rid` to correlate the reply with the call.

```javascript
const rid = Date.now().toString()
despia(`websocket://status?id=orders&rid=${rid}`)
```

```json
{
    "id": "orders",
    "type": "response",
    "ts": 1747600200,
    "rid": "1747600200000",
    "ok": true,
    "state": "open",
    "reconnectAttempt": 0,
    "pendingOutbound": 0,
    "unacked": 2
}
```

Every command accepts an optional `rid`, echoed back on the matching `response` event. Use it whenever you want a deterministic ack that a command was accepted, especially for `subscribe` and `send` which otherwise have no synchronous return.

A failed `response` carries `ok: false` and an `error` string. Failure cases are limited to malformed commands.

```json
{
    "id": "orders",
    "type": "response",
    "ts": 1747600200,
    "rid": "1747600200000",
    "ok": false,
    "error": "no_connection_for_id"
}
```

## Disconnecting

`websocket://disconnect` closes the socket and disables auto-reconnect. The durable store is preserved, calling `connect` with the same `id` reuses it.

```javascript
despia('websocket://disconnect?id=orders')
```

A clean disconnect fires a `closed` event with `code: 1001` and `clean: true`.

```json
{
    "id": "orders",
    "type": "closed",
    "ts": 1747600300,
    "code": 1001,
    "reason": "client_shutdown",
    "clean": true
}
```

To point an id at a different URL, disconnect first then issue a fresh `connect` with the new URL.

## Event reference

Every event includes `id` and `ts` (epoch seconds). Only `message` events participate in the ack flow.

| `type` | extra fields | durable |
| :-- | :-- | :-- |
| `connecting` | `attempt` | no |
| `open` | `protocol` | no |
| `message` | `message_id`, `dataType`, `payload` | yes |
| `reconnecting` | `attempt`, `delay`, `error` | no |
| `closed` | `code`, `reason`, `clean` | no |
| `error` | `error` | no |
| `response` | `rid`, `ok`, `error`, plus command result fields | no |
| `dropped` | `count`, `reason` | no |
| `send_failed` | `oid` | no |

`payload` is the parsed object or array for `dataType: "json"`, the raw string for `dataType: "text"`, and base64 for `dataType: "binary"`.

## Value reference

Closed value sets for command parameters and event fields. Anything not listed accepts any string.

| parameter | accepted values |
| :-- | :-- |
| `url` | string starting with `wss://` or `ws://` |
| `reconnect` | `"true"` (default), `"false"` |
| `protocols` | comma-separated subprotocol names, e.g. `graphql-transport-ws,graphql-ws` |
| `headers` | JSON-stringified object of HTTP header name and value pairs |

| event field | accepted values |
| :-- | :-- |
| `dataType` | `json`, `text`, `binary` |
| `state` | `connecting`, `open`, `waitingReconnect`, `closed` |
| `clean` | `true`, `false` |
| `ok` | `true`, `false` |
| `code` | numeric WebSocket close code per RFC 6455 |

When `ok` is `false` on a `response` event, `error` is one of:

| error | meaning |
| :-- | :-- |
| `missing id/url` | `connect` was missing `id` or `url` |
| `no_connection_for_id` | the targeted `id` has no live or registered connection |
| `unknown_command: <name>` | scheme host wasn't `connect`, `subscribe`, `send`, `disconnect`, or `status` |

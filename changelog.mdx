<Update label="March 26, 2026" description="Inline Ads - AdMob in your web UI, generally available">
  ## Inline Ads

  Despia now supports AdMob ads rendered directly inside your web UI as real DOM elements, not as native overlays above or below the WebView.

  This works by connecting Google's Mobile Ads SDK to the WebView (Android) or WKWebView (iOS) that Despia bundles inside your app. When your web content loads ad tags via AdSense, Google Publisher Tag, or IMA for HTML5, the SDK bridges app-level signals to improve ad relevance, fill rates, and revenue.

  **What ships today:**

  - **On-device SDK bridge.** `MobileAds.registerWebView()` (Android) and `MobileAds.shared.register()` (iOS) are called automatically during app startup. No JavaScript call is needed to activate the bridge.
  - **AdSense support.** Place standard `<ins>` ad units anywhere in your web UI. The SDK bridges app signals automatically.
  - **Google Publisher Tag support.** GPT slots work without modification once the WebView is registered.
  - **IMA for HTML5 video ads.** Pre-roll, mid-roll, and post-roll video ads play inline with no fullscreen takeover. Autoplay is supported on both Android and iOS without a user gesture.
  - **Safe area CSS variables.** `var(--safe-area-top)` and `var(--safe-area-bottom)` are injected automatically into every WebView, so top and bottom ad banners clear the notch, status bar, and home indicator.
  - **100% no-code setup.** Enable AdMob, paste your AdMob App ID, and rebuild from the Despia dashboard. No native code required.

  **Store compliance.** The SDK registers the WebView for ad signals only. No native code or executables are downloaded after submission. Compliant with Apple App Store Guideline 3.3.2 and the Google Play Malicious Behavior Policy.

  A rebuild is required after enabling Inline Ads in **Your App - Settings - AdMob**. The SDK registration cannot be applied over-the-air.

  **Documentation:** [Inline Ads](https://setup.despia.com/native-features/admob/inline-ads)
</Update>

<Update label="March 26, 2026" description="Share into app - cold start white screen fix">
  ## Share into app on cold start

  Share-into-app flows on cold start now navigate to the correct root screen instead of showing a white screen.

  Previously, when a user shared content into the app while it was fully closed, the share handler fired before the root navigation tree was mounted and function injection was ready. The result was a blank white screen that required a force-quit and retry to recover from.

  The fix holds the incoming share event until the root page is fully initialised and function injection is available, then delivers the payload and navigates correctly.

  **What changes:**

  - **Cold start share navigation.** The app now boots to a valid root screen when launched via a share sheet interaction. The share payload is delivered after initialisation completes.
  - **Function injection timing.** The share handler now waits for function injection to be ready before executing, eliminating the race condition that caused the blank screen.

  No dashboard configuration changes or integration code changes are required. The fix applies automatically to new deployments.
</Update>

<Update label="March 22, 2026" description="Local Server - generally available in beta">
  ## Local server

  Despia now ships a local server for iOS and Android. Your web build downloads to the device on first launch and is served from an on-device HTTP server at `http://localhost`. From that point on, the app boots with zero network latency and works completely offline.

  **What ships today:**

  - **On-device HTTP server.** Assets are served from `http://localhost`, not `file://` or a custom scheme. BrowserRouter, Vue Router, and any routing library that expects a real HTTP origin work without modification.
  - **First-launch hydration.** On first open, Despia fetches your web build from your existing hosting (Netlify, Vercel, AWS, or anything else) and caches it on the device. No migration required.
  - **OTA updates.** On startup, Despia fetches `despia/local.json` and compares the `deployed_at` timestamp with the cached value. If it has changed, the new build downloads in the background and applies on next launch. No app store review needed for HTML, CSS, JavaScript, image, or font changes.
  - **`@despia/local` build plugin.** Available for Vite, Webpack, Rollup, Nuxt, SvelteKit, Astro, Remix, esbuild, and any build system via a `postbuild` CLI hook. The plugin scans your output directory after each build and generates `despia/local.json` automatically.
  - **`despia-version-guard`.** A companion package for gating web UI features behind a minimum native runtime version. Supports React, Vue, Angular, Svelte, and Vanilla JS.

  **Store compliance.** The local server downloads web content only (HTML, CSS, JS, images, fonts). No native code or executables are downloaded after submission. The approach follows patterns established by Expo and is compliant with Apple App Store Guideline 3.3.2 and the Google Play Malicious Behavior Policy.

  **This feature is in final beta.** To request access, email [offlinemode@despia.com](mailto:offlinemode@despia.com).

  **Documentation:** [Local Server Introduction](/local-server/introduction) and [Reference](/local-server/reference).

  **Packages:**

  - [`@despia/local`](https://www.npmjs.com/package/@despia/local) — build plugin
  - `despia-native` — native SDK and JavaScript bridge
</Update>
